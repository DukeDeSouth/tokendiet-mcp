# Tokenizer — выбор и ограничения

**Решение (Sprint 0, 2026-07-11):** библиотека [`gpt-tokenizer`](https://www.npmjs.com/package/gpt-tokenizer) v3.x.

## Почему gpt-tokenizer

- Чистый JS/TS, без WASM и без загрузки rank-файлов по сети — работает офлайн, устанавливается без нативной сборки.
- Поддерживает оба нужных BPE-энкодинга через субпути (`gpt-tokenizer/encoding/o200k_base`, `.../cl100k_base`).
- Альтернатива `js-tiktoken` отклонена: требует отдельной поставки rank-данных, API менее удобен для дерева субпутей.

## Поддержанные энкодинги

| Энкодинг | Модели | Статус |
|---|---|---|
| `o200k_base` | GPT-4o, GPT-4.1, o1/o3 и новее | **дефолт** |
| `cl100k_base` | GPT-4, GPT-3.5-turbo, text-embedding-3 | опция |

Единица счёта одна на весь результат: `tokens_in` и `tokens_out` всегда считаются одним и тем же энкодингом.

## Ограничения (честное раскрытие)

1. **Anthropic (Claude).** Токенизатор Claude проприетарен и не опубликован. Для сессий Claude наш счёт по `o200k_base` — **приближение**: абсолютные числа могут отличаться, но *относительная экономия* (%) остаётся показательной, потому что числитель и знаменатель считаются одним энкодингом. Точная сверка через Anthropic `count_tokens` API — запланирована как замер в dogfood (Sprint 1), требует API-ключа.
2. **Gemini и другие.** Аналогично: приближение тем же способом.
3. **Никогда не используется chars/4** или другие символьные аппроксимации — ни в счётчике, ни в UI, ни в отчётах.

## Правило для кода

Весь подсчёт токенов идёт только через `src/tokenize/counter.ts` (`countTokens`). Прямой импорт `gpt-tokenizer` в других модулях запрещён — замена библиотеки должна оставаться локальной операцией.
