# Dogfood benchmark — TokenDiet Sprint 2 (v3)

**Date:** 2026-07-11  
**Status:** complete — v2 baselines (R1–R3) + v3 code/search/fetch scenarios  
**Harness:** `npm run dogfood:all` (includes R4, R5, SF) | gate summary: `npm run dogfood:v3-gate`

---

## Methodology (v3 additions)

### New scenarios

| # | Type | Description | Harness |
|---|------|-------------|---------|
| R4 | Code cold-first | 12 outline reads on `src/pipeline/**`, tools, tests | `npm run dogfood:r4` |
| R5 | Code intra-session | outline + symbol drill-down + re-read + test run | `npm run dogfood:r5` |
| SF | Search + fetch | 5 repo searches + 3 live URLs (MDN, GitHub API, README) | `npm run dogfood:sf` |

### Harness extensions (`scripts/dogfood-lib.ts`)

- `benchReadMode(ctx, path, mode, label?, symbol?)` — explicit `outline` / `symbol` / `auto`
- `benchSearch(ctx, query, label?, glob?)` — baseline = raw rg output tokens
- `benchFetch(ctx, url, label?)` — baseline = raw HTTP body tokens

### Accounting

Same rules as v2 (`benchmarks/2026-07-11-dogfood.md`). Cold-first = first read in fresh MCP process with no prior `servedThisSession`.

## Notes

- R4/R5 use `mode=outline` explicitly — models agent behavior after Sprint 2 descriptor guidance.
- SF fetch requires network; GitHub API and plain README pass through uncompressed (expected).
- Re-read **outline → outline** with same hash → `unchanged` (0 tokens out) when `hasSufficientDetail` passes (`readMode.ts`; test: `returns unchanged on second outline read with same hash`). R5’s final re-read follows a `symbol` serve on the same file, so outline is re-compressed — expected, not a harness bug.
- USD figures: same `pricing.json` as v2.
